# PWA 环境优先重构需求整理

更新时间：2026-06-05

## 文档目的

本文档记录 2026-05-27 关于 Pokemon Champions Tool 下一阶段界面与功能层级调整的需求共识。它不是最终高保真原型，也不是实现计划，而是给后续 OpenDesign 原型与开发计划使用的需求底稿。

本轮讨论的核心变化是：应用默认入口从“队伍 / 组队”调整为“环境”，让浅度用户打开后先获得当前赛季环境价值；队伍、计算器、速度线、图鉴等现有功能继续保留，但入口与层级需要重新组织。

实现状态备注（2026-06-05）：环境页已接入 PokeDB Season 1 独立 JSON snapshot，当前可展示 Pokémon 样本占比、常用招式、携带道具占比、常见队友占比与首批真实队报样本。常用招式不是 Open Data 字段，而是维护脚本解析 Pokémon 详情页 `data-move-detail` 生成的 `moveStats`，当前覆盖单打 / 双打各前 50 Pokémon；队报样本来自 PokeDB trainer/list 页面解析，当前单打 / 双打各保留 8 条完整队伍槽位样本和外部队报链接。snapshot 运行时从 `public/data/pokedb/reg-ma-s1-environment.json` 加载，并由 service worker 预缓存。

## 重构前项目对齐

2026-05-27 梳理需求时，项目已经具备以下能力：

- 移动端优先的 React + Vite + TypeScript + Tailwind PWA 壳。
- 底部 5 Tab：组队、计算、速度线、图鉴、设置。
- 本地 IndexedDB 队伍管理，支持多队伍、成员编辑、JSON 导入 / 导出。
- 当前规则图鉴，包含 Pokemon、招式、道具、特性、Mega 信息。
- 速度线工具。
- 实验性伤害计算器。
- 浅色 / 深色主题、PWA 离线能力、移动端视觉回归测试基础。

本轮规划不是推翻现有功能，而是重排入口：

- 原“组队”能力调整为新的“队伍”页。
- 原“计算 / 速度线 / 图鉴”能力收束为新的“工具”页入口组。
- 新增“环境”作为默认首页。
- 原“设置”能力与收藏、历史、数据来源说明等归入“我的”。

### 现有 PWA 界面参考核对

根据现有代码与当前截图，重构应继承现有移动端工具风格，而不是重做成另一套普通表单页面：

- `App.tsx` 当前使用 430px 移动端外壳、固定底部导航、全局 Header。重构后底部导航从 5 Tab 调整为 4 Tab：环境、队伍、工具、我的。
- `Header.tsx` 当前展示 `Champions Tool` 与 `Regulation Set M-A · 移动端 MVP`，说明规则上下文已经是全局信息的一部分。新环境首页可承接更完整的规则 / 赛季 / 数据来源信息。
- `BottomNav.tsx` 当前是固定底部导航，图标 + 文案 + 激活点。新导航应保留这种移动端稳定锚点，只调整入口数量与命名。
- `ui.tsx` 已有 `Card`、`Button`、`Chip`、`Badge`、`TypeBadge`、`PokemonAvatar` 等组件。新页面草图应优先复用这种深色卡片、属性胶囊、Pokemon 头像和紧凑按钮语言。
- `SpeedPage.tsx` 已有 Pokemon / 形态选择、SP 滑杆、最终速度卡、速度散点图、当前队伍 / 收藏 / 常见 / 全部 Tab、benchmark 列表与详情 Sheet。新的“速度线计算”入口应进入这个能力，而不是重做简化版。
- `CalculatorPage.tsx` 已有进攻方 / 防守方卡片、搜索图鉴、从队伍选择、能力配置折叠区、性格 / 特性 / 道具 / 招式 / SP / 阶级、天气 / 场地 / 单双打规则与结果卡。新的“伤害计算”入口应保留这种卡片式配置体验。
- `DexPage.tsx` 已有 Pokemon / 招式 / 道具 / 特性 Tab、搜索、属性筛选、Mega 筛选、Pokemon 详情和跳转速度线 / 伤害计算能力。新的“规则图鉴”入口应承接现有图鉴页面。

因此，“工具页”不是替代现有三个工具页面，而是把原来底部导航里的计算、速度线、图鉴收束为一个入口聚合页。进入具体工具后，继续沿用现有页面逻辑与视觉资产。

## 还有多少没梳理

当前需求大约完成了主体产品方向梳理。已经明确的是产品边界、主导航、环境首页优先级、队伍页定位、高分队伍导入、队报链接、生成图片、工具页、我的页与主要状态逻辑。

仍待梳理的主要部分：

- PokeDB 数据接入后的字段边界：哪些来自 Open Data，哪些来自页面解析，哪些需要人工补充或暂不承诺。
- 队伍导入后的命名规则：用队报标题、作者、分数、排名、核心 Pokemon 还是混合生成。
- 伤害计算、速度线计算、规则图鉴进入工具页后的二级页面是否需要微调视觉与入口文案。
- 首版实现优先级：先做环境首页 + 高分队伍导入，还是同时调整底部导航与队伍页。

## 已确认需求

### 1. 产品边界

Pokemon Champions Tool 不是一个强引导式组队推荐工具。它面向已经有一定对战理解的用户，帮助他们查看环境、保存配置、验证配置、复盘队伍，而不是要求用户必须从零开始接受推荐。

页面文案应避免命令式和强推荐感。推荐使用：

- 参考
- 对照
- 保存
- 验证
- 复盘
- 导入配置
- 常同队出现
- 样本占比

避免使用：

- 最佳队友
- 推荐搭档
- 你应该
- 必须补强
- 官方使用率

### 2. 新底部导航

底部导航调整为 4 个主入口：

- 环境
- 队伍
- 工具
- 我的

其中“环境”为默认首页。“队伍”为第二个 Tab，承接用户本地队伍管理与高分队伍导入后的本地副本。“工具”收纳伤害计算、速度线计算、规则图鉴三个并列入口。“我的”收纳设置、收藏、历史、数据来源说明与缓存管理。

### 3. 环境首页作为默认入口

环境首页面向浅度用户和高频回访用户。用户即使没有创建队伍，也应该能打开应用获得价值。

首页不应强绑定“我的队伍”。“我的队伍对照”不进入环境首页首版，避免浅度用户打开环境页时被迫进入个性化队伍流程。

环境首页首屏优先回答三个问题：

- 现在环境最热的是什么？
- 这些热门对象通常怎么组成队？
- 我可以拿这些信息做什么？

### 4. 环境首页信息优先级

环境首页模块优先级：

1. 热门宝可梦榜
2. 热门队伍样本
3. 常同队出现 / 热门核心

“热门宝可梦榜”必须突出。它是首页的主模块，而不是普通列表入口。

首页不做全局道具榜。道具总榜在当前规则下信息价值有限，而且 Mega Stone 等道具与特定 Pokemon 强绑定，做全局榜容易产生误解。道具数据下沉到宝可梦环境详情页，作为“这只 Pokemon 的常见携带道具”展示。

首页不做全局热门技能榜。常用招式只放在特定 Pokemon 的环境详情页中，因为招式价值必须绑定具体 Pokemon 才有解释意义。

“热门核心 / 常见队友”需要弱化。它不应作为首页强推荐模块，而应放在宝可梦详情页或折叠模块中，文案使用“常同队出现”或“样本队伍中共同出现”。

### 5. 环境首页筛选

环境首页顶部不做重筛选栏。首页首屏只展示当前正在查看的数据上下文，避免用户打开后先面对一组复杂选择。

首页顶部常驻信息：

- 当前赛季，例如 `Season 1`。
- 当前规则，例如 `单打` 或 `双打`。
- 数据更新时间。
- 单打 / 双打胶囊切换。

首页不使用“筛选”按钮作为单打 / 双打切换入口。单打 / 双打应使用有辨识度的胶囊切换控件，放在环境标题下方或规则上下文附近。

完整筛选项可以作为后续筛选层或底部弹层：

- 赛季切换。
- 规则切换。
- 单打 / 双打切换。
- 数据来源说明。
- 更新时间。

推荐首页顶部文案：

```text
环境
Season 1 · 单打
基于高分构筑样本统计 · 更新于 05-27 10:00
```

推荐筛选层文案：

```text
环境筛选
赛季：Season 1
规则：单打 / 双打
数据来源：PokeDB 高分构筑样本
更新时间：2026-05-27 10:00
```

### 6. 环境榜单区域

环境首页的主榜单区域只保留“宝可梦榜”，不再将“道具榜”与“宝可梦榜”并列展示。

原因：

- 当前规则下可用道具数量有限，道具总榜的信息价值弱于宝可梦榜。
- 道具存在强绑定关系，例如 Mega Stone 只对特定 Pokemon 有意义，做全局总榜容易产生误导。
- 道具更适合作为特定 Pokemon 的详情数据，例如“这只 Pokemon 常见携带道具”。

首页榜单默认展示 Top 10。前几名可以有轻微视觉强调，但不需要做成复杂大卡。榜单底部提供“查看全部”，进入完整宝可梦榜单页。

完整榜单页用于承载更多数据，例如 Top 50 或全部样本统计，并保留赛季 / 规则筛选。

完整榜单页需要与环境首页保持同一套全局规则切换视觉。首页使用的“单打 / 双打”胶囊开关，在完整榜单页中也应保持一致，不再拆成两个独立胶囊。

“数据来源”不是与“单打 / 双打”并列的规则选项，不应使用同级胶囊。它应作为弱文本链接或说明入口展示在更新时间附近。

完整榜单页不向用户展示内部规则说明，例如“显示 Top 50”“点击条目查看详情”。展示数量和点击行为由产品默认规则决定，不需要额外解释。

### 7. 热门宝可梦榜字段

热门宝可梦榜建议字段：

- 排名
- Pokemon 头像
- Pokemon 名称与形态
- 属性图片 / 属性标签
- 比重，即样本占比 / 高分样本出现率
- 出现队伍数

榜单不建议直接使用“使用率”作为主文案，除非后续确认数据能够代表完整游戏环境。当前更准确的表述是“样本占比”或“高分样本出现率”。

宝可梦榜卡片内不展示道具。道具信息进入宝可梦环境详情页，避免首页榜单卡片字段过多。

点击榜单中的 Pokemon 后进入新的“宝可梦环境详情页”，而不是跳转到原有图鉴页。原有图鉴页继续保留，用于百科式查询；环境榜单详情页用于展示当前赛季对战环境相关数据。

### 8. 特定 Pokemon 的携带道具

道具不作为环境首页全局榜单主入口。道具数据下沉到宝可梦环境详情页，展示“当前 Pokemon 的常见携带道具”。

特定 Pokemon 的携带道具模块可以展示：

- 排名
- 道具图标
- 道具名称
- 比重，即样本占比 / 出现次数
- 出现队伍数

首版不做全局道具详情页。用户如果关心道具，应先从宝可梦详情页理解“这只 Pokemon 通常带什么”。

### 9. 宝可梦详情页定位

宝可梦详情页不应做成传统百科图鉴页。现有图鉴页已经承担百科信息查询，环境详情页的价值应是解释当前环境：

- 这只 Pokemon 在当前高分样本中的热度如何？
- 常带什么道具？
- 常和哪些 Pokemon 同队出现？
- 有哪些真实高分队伍可参考？

宝可梦详情页建议模块顺序：

1. 顶部概览
2. 常用招式
3. 常见道具
4. 常同队出现
5. 相关高分队伍
6. 后续扩展配置统计

其中“常同队出现”需要使用中性文案，例如“高分样本中常同队出现”，避免被误解为强推荐。

宝可梦环境详情页需要控制层级深度，避免 PWA 变成过重的多级 App。招式和道具可以展示详情，但不再进入新的三级页面。

交互原则：

- 常用招式展示 Top 10。
- 常见携带道具展示 Top 10。
- 常同队出现展示 Top 7。
- 详情页模块默认展示 Top 3，用户点击“查看全部”后在当前模块位置直接页内展开。
- 不使用底部抽屉承载 Top10 / Top7 列表，避免用户视线从模块上方跳到底部，也避免全面屏机型底部手势冲突。
- 点击招式或道具时，使用当前条目的页内展开态展示详情，不跳转到新的完整页面。
- 点击常同队 Pokemon 可以进入该 Pokemon 的环境详情页，但需要保留返回路径。
- 相关高分队伍中的“导入配置”和“队报链接”仍作为明确动作按钮。

招式详情可展示：

- 招式类型。
- 物理 / 特殊 / 变化分类。
- 威力。
- 命中。
- 标签，例如接触、优先度。
- 简短效果说明。

道具详情可展示：

- 道具图标。
- 道具名称。
- 携带比重。
- 出现队伍数。
- 简短效果说明，如果本地 catalog 中已有。

首版宝可梦环境详情页应以可稳定获得的数据为主。基于 PokeDB Open Data，首版可稳定展示：

- 当前规则下排名。
- 样本占比。
- 出现队伍数。
- 最高排名 / 最高 rating。
- 常见道具分布。
- 常同队出现分布。
- 相关高分队伍样本。

常用招式数据不在 PokeDB Open Data 里，但 PokeDB 宝可梦详情页的 HTML 中存在结构化招式统计，例如 `data-move-detail` 暴露 `rank`、`move_key`、`name`、`rate`、`type`、`category`、`power`、`accuracy`、`tags`、`description` 等字段；页面中还存在 `pokemonShowTrend` 的 moves 趋势数据。因此技术上可以通过服务端定时抓取详情页并解析缓存接入。

常用招式的产品价值较高。原型可以把“常用招式”放在宝可梦环境详情页首屏核心模块，但需求文档需要标注：

- 它不是 Open Data 稳定字段。
- 不能让终端用户设备直接请求 PokeDB 页面。
- 需要由我们自己的服务端定时抓取、解析、缓存。
- 首版开发前需要做解析稳定性验证和合规边界确认。

首版可以结合项目本地数据展示，但要明确它们不是环境统计：

- 属性。
- 形态。
- 基础图鉴信息。
- 速度线入口。
- 伤害计算入口。

以下数据暂不作为首版强承诺，因为它们不在 PokeDB Open Data 的稳定字段里，可能需要解析页面或额外来源：

- 常见性格。
- 常见特性。
- 努力值 / SP 配置。
- 完整配招组合。

如果后续确认页面解析稳定且合规，可以在详情页增加“配置趋势”模块，但首版原型中应以灰度占位或后续能力表达，不把它画成已确定能力。

### 10. 高分队伍样本

高分队伍样本是环境数据转化为用户行动的重要模块。它可以出现在环境首页、宝可梦详情页，以及后续专门的高分队伍浏览页中。

队伍卡片展示：

- 作者
- 分数 / 排名
- 队伍标题或核心描述
- 6 只 Pokemon 头像
- 来源标识
- 两个按钮：导入配置、队报链接

“导入配置”会把该高分队伍复制为用户本地队伍，并跳转到“队伍”页。

导入后的落点已确认：跳转到队伍列表，并高亮新导入的队伍卡片。不要直接进入队伍详情或编辑页。

原因：

- 用户需要先确认配置已经进入自己的本地队伍列表。
- 队伍列表能保留“这是一个本地副本”的心理模型。
- 用户可以在队伍列表中选择继续编辑、生成图片，或稍后再处理。

“队报链接”直接打开外部原文，不进入中间详情页。队报在宝可梦社区语境里代表队伍所有者自己写的构筑拆解和对战思路文章，因此按钮描述必须准确。

### 11. 队伍页定位

队伍页默认显示队伍列表，而不是直接进入某个队伍详情。这样可以容纳更多用户本地队伍，也能自然承接高分队伍导入后的本地副本。

队伍来源可能包括：

- 用户手动创建
- 从高分队伍导入
- 后续从队报或外部链接导入

不论来源如何，进入队伍页后卡片样式应保持一致。来源只作为标签或元信息出现，不改变主要按钮。

队伍卡片按钮统一为：

- 编辑配置
- 生成图片

### 12. 生成图片

“生成图片”是队伍页的轻量分享能力，不做复杂导出。

交互已确认：

- 点击“生成图片”后直接生成默认队伍图。
- 不进入模板选择页。
- 不选择字段。
- 生成完成后弹出轻量结果层。
- 结果层只展示图片和“保存图片”按钮。
- 不提供“分享”按钮。

默认队伍图内容：

- 队伍名称
- 6 只 Pokemon
- Pokemon 图片
- Pokemon 名称
- 属性
- 性格
- 特性
- 道具图标
- 道具名称
- 配招
- 生成时间戳

视觉参考方向：

- 类似 Cardman / Party Card 风格的黑底 2x3 队伍图。
- 重点是配置可读性，不是训练师卡片。
- 每个 Pokemon 占一个格子，保证手机端保存后仍能看清。

### 13. 队伍导入后的命名

导入配置后，系统应自动生成队伍名称，并允许用户后续编辑。

命名规则待确认，但方向可以包含：

- 原作者
- 分数 / 排名
- 队报标题
- 核心 Pokemon
- 玩法标签

示例：

- `作者名 · 2724 · 喷火龙 + 风妖精`
- `Rank 1 · 高速晴天 · 喷火龙核心`
- `队报标题 · 作者名`

### 14. 数据来源与表达边界

环境数据应明确标注来源和更新时间。当前表达不应包装成官方完整使用率。

建议文案：

- 基于高分构筑样本统计
- 样本占比
- 出现队伍数
- 数据更新时间
- 数据来源说明

不建议文案：

- 官方使用率
- 全环境使用率
- 最强推荐
- 必带

PokeDB Open Data 可以作为首阶段稳定来源，优先支持：

- 高分队伍样本
- Pokemon 出现统计
- 道具出现统计
- 队友共现统计

技能、特性、性格、配招、努力值等细节如果需要从详情页解析，应作为后续阶段，不在首版首页强承诺。

## 待确认需求清单

### 1. 是否需要独立“高分队伍”页面

目前高分队伍样本会出现在环境首页和宝可梦详情页，但是否需要独立浏览页仍未确认。

如果高分队伍数量较多，建议后续增加独立页面，支持按 Pokemon、道具、分数、规则筛选。

### 4. 道具详情页是否首版需要

首版不做全局热门道具榜，也不把道具详情页作为环境首页入口。道具数据下沉到宝可梦环境详情页中的“常见携带道具”模块。

后续如果要做道具详情页，建议从宝可梦环境详情页或相关队伍筛选结果进入，而不是从首页全局道具榜进入。

### 5. 队伍详情页的信息结构

队伍列表已确认，点击“编辑配置”进入队伍详情 / 编辑入口页。队伍详情页定位为“我的本地队伍工作台”，不再强调来源。

- 6 个成员采用现有 PWA 队伍页卡片样式：Pokemon 图片、名称、属性、道具图标、道具名称。
- 成员卡片不直接展示招式，避免队伍详情页过密；点开成员编辑后再展示招式。
- 编辑配置仍沿用现有成员编辑流程。
- 成员编辑页沿用当前 PWA 的底部 Sheet 结构，不改成新的“基础配置卡片网格”。结构为：顶部编辑标题与关闭按钮、成员摘要卡、特性选择、道具搜索与列表、性格选择、4 个招式位、SP 分配、校验结果、底部删除 / 取消 / 保存。
- 道具选择继续保留搜索框和可选道具列表，列表项展示道具图片、名称、效果说明。
- 招式仍按 4 个招式位展示，点开某个招式位后在该位内搜索和选择。
- SP 单项点击后继续使用独立调整器或轻量选择层。
- 队伍详情里未满 6 人时显示空槽位 / 添加入口。点击空槽先打开 Pokemon 选择 Sheet，选中后建议直接进入成员编辑 Sheet，而不是先回到队伍详情再让用户二次点击。
- 已有成员编辑时，摘要卡保留 Pokemon 图片、名称、属性、道具。若需要更换 Pokemon，可在摘要卡右侧提供轻量“更换”入口，回到 Pokemon 选择 Sheet 后继续使用同一个成员编辑流程与保存校验。
- 队伍详情页不放“生成图片 / 伤害计算 / 速度线”快捷跳转。它们偏小工具性质，会打断队伍编辑流程。
- “生成图片”保留在队伍列表卡片上。
- “伤害计算 / 速度线”保留在工具页，并在工具页内提供从队伍导入配置的渠道。
- 来源为高分队伍时，不在队伍详情页单独展示来源卡片或队报链接。用户导入前已经可以通过环境页打开队报；导入后该队伍是本地副本，队伍名称已经包含作者 / 分数 / 核心等来源信息。
- 队伍改名不放在更多菜单里。用户点击队伍标题即可进入可编辑文本框。
- 不使用 `...` 更多菜单。
- 删除队伍放在队伍详情页底部危险区，不放在顶部，避免误触，也避免标题区变复杂。

### 6. 工具页具体层级

工具页需要收纳现有三个并列工具入口：

- 伤害计算。
- 速度线计算。
- 规则图鉴。

工具页定位为“临时计算与查询中心”。用户不需要先创建队伍也能直接使用伤害计算、速度线计算和规则图鉴。

工具首页不展示“队伍配置带入”说明模块，也不放解释性注释。导入队伍配置是具体工具内部的字段动作，例如伤害计算里的“从队伍选择攻击方 / 防守方”、速度线计算里的“从队伍生成速度线”。

导入队伍配置只作为具体工具页的临时预填，不自动写回队伍。用户在伤害计算或速度线中修改参数时，默认不会改变本地队伍配置，避免“查一下”变成误改队伍。

工具页不预设“常用 / 查询”的优先级，三个入口平级展示。当前规则不再作为工具页入口，规则上下文放在环境页与数据来源说明中。

工具页内仍需保留从队伍带入的能力，例如：

- 从队伍选攻击方。
- 从队伍选防守方。
- 从队伍生成速度线。

### 7. 我的页范围

“我的”页是低频管理页，承接原设置能力，但不抢环境、队伍、工具三个主流程。

首版建议承接：

- 主题设置。
- 语言偏好。
- 本地资料备份 / 恢复。
- 收藏入口。
- 缓存 / 离线状态。
- 数据来源说明。
- 清除本地数据。

“当前规则”不作为“我的”页强入口。规则、赛季、来源、更新时间主要跟随环境页展示；“我的”里只保留更底层的数据来源说明与缓存状态。

“历史”可以作为后续能力，首版不强行加入，避免“我的”页膨胀成杂物页。

“我的”页不展示本地队伍数、收藏数这类低价值概览数字，避免首屏被弱信息占用。

需要区分两种导出：

- 队伍页的“生成图片”：用于分享展示，只针对单支队伍，输出队伍名称、成员图片、属性、道具、配招等视觉信息。
- “我的”页的“导出备份 JSON”：用于迁移和备份，应包含本地队伍、收藏、偏好设置、缓存索引等本地资料，而不是只导出队伍信息。

如果后续在“我的”页提供图片导出入口，也应明确是跳转到队伍页选择具体队伍生成图片，不与备份 JSON 混在一个动作里。

### 8. 离线体验

当前项目已有 PWA 离线能力。环境数据接入后，状态策略如下：

- 数据正常：展示环境榜单与热门队伍样本，标注赛季、规则、数据来源与更新时间。
- 数据加载中：保留页面骨架，不跳转到其他页；榜单和队伍样本显示骨架屏或轻量 loading。
- 数据过期但可用：继续展示缓存内容，在标题或更新时间附近提示“使用上次缓存 / 更新于 xx”，不使用大面积错误页。
- 离线但有缓存：继续展示上次缓存内容，弱提示“离线模式 · 使用上次缓存”。
- 离线且无缓存：展示空状态“无法加载环境数据”，提供“重试”。这是环境页唯一需要阻断内容的场景。
- 部分字段缺失：能展示的字段继续展示，缺失字段不占位造假；涉及队伍导入时走导入缺失配置策略。

只要本地有可用缓存，就不应阻断用户浏览环境页。环境首页是默认入口，错误状态必须克制，避免用户一打开应用就看到整页失败。

离线状态下是否允许导入高分队伍取决于本地缓存是否包含足够配置字段：

- 缓存完整：允许导入。
- 缓存只含队伍骨架：允许导入，但标记为需要补全。
- 缓存缺少 Pokemon 或队伍成员核心字段：阻止导入。

### 8.1 高分队伍导入状态

高分队伍导入需要区分三类结果：

- 导入成功：Toast 显示“已导入配置”，跳转到队伍列表，并高亮新导入队伍 2-3 秒。
- 导入成功但字段缺失：Toast 显示“已导入，可继续补全配置”，同样跳转队伍列表。队伍详情 / 成员编辑中通过现有校验展示缺失项。
- 导入失败：展示轻量错误层或 Toast，文案为“导入失败，这支队伍缺少必要配置，暂时无法导入”。保留用户在当前高分队伍页，不跳转。

导入不应因为小字段缺失而失败。只要能形成本地队伍骨架，就允许进入队伍列表；缺失配置交给队伍校验和成员编辑处理。只有完全无法识别 Pokemon、队伍为空、规则不匹配到无法转换等情况才阻止导入。

### 9. 数据合规与授权边界

需要继续确认：

- PokeDB Open Data 的使用方式与频率。
- 是否允许服务端定期抓取并缓存。
- 队报原文链接是否只保存 URL，不抓取全文。
- 是否需要在数据来源页展示来源、更新时间与免责声明。

## 给 OpenDesign 的需求摘要

请为 Pokemon Champions Tool 设计一版移动端 PWA 信息架构重构原型。目标不是营销页，而是实际可用工具界面。应用面向有一定宝可梦对战理解的用户，核心价值是查看当前环境、参考高分队伍、保存和编辑队伍配置、使用计算工具验证思路。

底部导航调整为 4 个入口：环境、队伍、工具、我的。默认进入“环境”。环境首页首屏突出“热门宝可梦榜”，辅助展示“热门队伍样本”。道具和招式不做全局榜，放入特定 Pokemon 的环境详情页。环境数据表达为“基于高分构筑样本统计”，避免使用“官方使用率”等过度承诺文案。

“队伍”页默认显示队伍列表。队伍可以来自用户手动创建，也可以从高分队伍导入。不同来源的队伍卡片样式保持一致，只通过标签展示来源。队伍卡片提供“编辑配置”和“生成图片”两个按钮。“生成图片”点击后直接生成默认黑底 2x3 队伍分享图，弹出结果层，只提供“保存图片”按钮。

“工具”页收纳现有伤害计算、速度线计算、规则图鉴三个并列入口，不再放当前规则入口。“我的”页收纳主题、语言、备份 / 恢复、收藏、缓存、数据来源说明和清除本地数据等设置类内容。

整体界面应移动端优先，信息密度高但不拥挤，避免营销式大 hero 和过重装饰。风格应专业、清晰、适合反复查询。
